# MinimalWebKitBrowser

A minimal, highly customizable, extremely snappy WebKit browser for Apple Silicon Macs (M1+).

Built with native Swift + SwiftUI + WebKit (`WKWebView`). Targets macOS 14 Sonoma and later. Optimized for the lowest possible RAM/CPU and the most responsive tab switching — noticeably lighter than Safari or Orion for users who want maximum control and minimal overhead.

> Tabs live in a **Zed-style collapsible side sidebar**, not a horizontal top bar.

## Build

The project is generated from `project.yml` with [XcodeGen](https://github.com/yonaskolb/XcodeGen).

```bash
# 1. install XcodeGen (one time)
brew install xcodegen

# 2. regenerate the Xcode project
cd browsermaking
xcodegen generate

# 3. build & run in Xcode
open MinimalWebKitBrowser.xcodeproj
#   select the "MinimalWebKitBrowser" scheme → ⌘R
```

Command-line build (requires full Xcode, not just CLT):

```bash
xcodebuild -project MinimalWebKitBrowser.xcodeproj \
           -scheme MinimalWebKitBrowser \
           -configuration Release \
           -destination 'platform=macOS' build
```

Requirements: macOS 14+, Xcode 15+, Apple Silicon (arm64) recommended.

## Why it feels snappy

| Lever | Where | Effect |
|---|---|---|
| `WKPreferences.InactiveSchedulingPolicy` | `WebViewConfig` | throttles/suspends JS+layout on idle tabs (macOS 14+) |
| Lazy tab creation | `Tab.loadIfNeeded()` | background/restored tabs cost ~0 until shown |
| Coarse unloading | `TabManager.runUnloadPass()` | tears down stale `WKWebView`s after N min / over cap |
| Reparenting host | `WebViewHost` | one persistent container; tab switch = show/hide, no reload |
| KVO per-tab | `Tab.installObservers` | only the active tab's progress re-renders |
| No process-pool fiddling | `WebViewConfig` | `WKProcessPool` is a no-op since macOS 12; WebKit owns it |
| LTO + dead-code-strip | `Release.xcconfig` | leaner binary, faster launch |

## Tabs — Zed-style collapsible sidebar

- Vertical sidebar on the **left** (default) or **right**, never a horizontal bar.
- **Collapse/expand**: click the chevron in the sidebar header, or `⌘⌥S`.
  - Collapsed → ~48pt icon-only rail (per-tab glyph + active indicator bar).
  - Expanded → full width with titles + hover-revealed close buttons.
- **Resize**: drag the inner edge (160–420pt). Width persists.
- **Show/hide entirely**: `⌘⇧S`, or disable in Settings → Appearance.
- Active tab has an accent indicator on its leading edge (Zed-style).

## Customize

Three layers, all synced:

### 1. Settings window (`⌘,`)
General · Performance · Appearance · Privacy · Advanced — every control binds directly to the store and auto-persists.

### 2. JSON config file
Power-user config at:

```
~/.config/minimal-webkit-browser/config.json
```

Created automatically on first launch with every key present. Edit it by hand; the app reloads on next launch and fills any missing keys with defaults (forward/backward compatible). Use **Settings → Advanced → Reveal config.json** to open it in Finder.

Key settings:

```jsonc
{
  "homepageURL": "https://duckduckgo.com",
  "searchEngineURL": "https://duckduckgo.com/?q=%@&ia=web",
  "newTabPageURL": "",                 // "" = local blank page
  "restoreLastSession": true,

  "tabUnloadMinutes": 15,              // 0 = never tear down idle tabs
  "maxLoadedTabs": 12,                 // 0 = unlimited
  "inactivePolicy": "throttle",        // suspend | throttle | none

  "tabPosition": "left",               // left | right | hidden
  "sidebarVisible": true,
  "sidebarCollapsed": false,
  "sidebarWidth": 230,
  "compactMode": false,
  "theme": "system",                   // system | light | dark
  "pageZoom": 1.0,

  "enableJavaScript": true,
  "blockTrackers": true,
  "privateBrowsing": false,

  "customUserAgent": "",
  "injectCustomCSS": false,
  "customCSS": "body { background: #000 !important; }",
  "injectCustomJS": false,
  "customJS": "console.log('hello from MinimalWebKitBrowser');"
}
```

### 3. User script / CSS injection
Enable in Settings → Advanced (or the JSON keys above). Custom CSS is injected as a `<style>` element at document end; custom JS runs at document end. Both apply to every frame on every navigation.

## Privacy

- Tracker/ad blocking via a bundled `WKContentRuleList` (`blocklist.json`).
- HTTPS upgrade for known HSTS hosts, fraudulent-site warnings, optional private browsing (non-persistent data store).
- Disable any of it in Settings → Privacy or the JSON file.

## Extending it

The architecture is intentionally modular:

- **New tab-level feature** → add logic in `Tab` + a `TabContext` method, wire through `TabManager`.
- **New setting** → add a field to `BrowserSettings` (with a `CodingKey` + `dec()` line), expose it in `SettingsView`, consume it where needed.
- **New sidebar mode / panel** → branch in `SidebarTabs` / `BrowserWindow`.
- **More blocking rules** → edit `DefaultContentBlocker/blocklist.json` (rebuild recompiles).
